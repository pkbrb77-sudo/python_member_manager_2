# CLAUDE.md — 프로젝트 가이드 및 규칙

## 1. 소개 (Introduction)
본 프로젝트는 효율적이고 안정적인 **연락처 관리 웹 서비스(Contact Management Web Service)**를 구축하는 것을 목표로 합니다. 시스템 아키텍처, 데이터베이스 스키마, 엄격한 API 명세 및 에러 핸들링 전략을 기반으로 하며, 모든 개발자가 일관된 방향성을 유지하고 원활하게 협업할 수 있도록 돕습니다.

---

## 2. 규칙 (Rules — 어떻게 해야 하는지)
프로젝트의 코드 품질과 일관성을 위해 다음 개발 규칙을 반드시 준수해야 합니다.

* **코드 스타일 및 일관성:**
  * 모든 코드는 프로젝트에서 지정한 표준 린터(Linter)와 포매터(Formatter) 규칙을 통과해야 합니다.
  * 명확하고 의도가 드러나는 변수명, 함수명을 사용하며 기술 스택의 베스트 프랙티스를 따릅니다.
* **아키텍처 및 데이터 흐름:**
  * 정의된 시스템 아키텍처와 레이어드 패턴(Layered Architecture)을 엄격히 준수합니다.
  * 데이터베이스 스키마 변경이 필요한 경우, 반드시 마이그레이션 스크립트를 작성하고 팀 내 사전 리뷰를 거쳐야 합니다.
* **API 및 에러 핸들링:**
  * 새로운 API를 추가할 때는 TRD의 API 명세서 형식을 그대로 유지하며, 엔드포인트와 입출력 타입을 명확히 정의합니다.
  * 모든 예외 상황은 사전에 정의된 에러 핸들링 전략(공통 에러 응답 포맷)에 맞추어 처리해야 합니다.
* **Git 및 협업 절차:**
  * 기능 개발 시 작업 단위별로 브랜치를 분기하며, 커밋 메시지는 규칙(예: `feat:`, `fix:`, `docs:`)에 맞게 작성합니다.
  * 모든 코드는 메인 브랜치 병합 전, 구현 체크리스트를 만족하고 최소 1명 이상의 리뷰어에게 승인을 받아야 합니다.

## 3. 금지 (Prohibitions — 무엇을 하면 안 되는지)
안정적인 서비스 운영과 결함 방지를 위해 아래 사항들을 엄격히 금지합니다.

### 3.1. 에러 핸들링 및 예외 우회 금지
* **임의 객체 및 텍스트 반환 금지:** API 호출 실패나 내부 로직 에러 발생 시, 사전에 약속된 공통 에러 응답 포맷(예: `{ success: false, error: { code: '...', message: '...' } }`)을 우회하고 임의의 일반 텍스트나 빈 객체(`{}`), 혹은 `null`을 반환하는 행위를 금지합니다.
* **실패(Catch) 블록 방치 금지:** `try-catch` 문을 사용할 때, 예외를 포착한 뒤 아무런 처리도 하지 않고 비워두거나(`catch (e) {}`), 단순히 콘솔 로그(`console.error(e)`)만 남긴 채 프로세스를 정상 흐름으로 진행시키는 행위를 엄격히 금지합니다. 모든 에러는 반드시 로깅 후 적절한 예외 처리 메커니즘을 타야 합니다.

### 3.2. 보안 및 환경 설정 위반 금지
* **민감 정보 하드코딩 절대 금지:** 데이터베이스 접속 자격 증명(ID/PW), API 시크릿 키, JWT 서명 키, 외부 서비스 토큰 등의 민감한 정보(Credentials/Secrets)를 소스 코드 내에 문자열로 직접 하드코딩하는 것을 금지합니다. 모든 설정값은 반드시 `.env` 파일과 환경 변수를 통해 주입받아야 하며, 시스템 설정 파일(`config.js` 등)을 통해서만 접근해야 합니다.
* **개발 편의성용 보안 우회 금지:** 로컬 개발이나 테스트 편의를 위해 CORS(Cross-Origin Resource Sharing) 설정을 와일드카드(`*`)로 무조건 허용하거나, 인증/인가(Authentication/Authorization) 미들웨어를 임의로 주석 처리하여 커밋하는 행위를 금지합니다.

### 3.3. 독단적인 스키마 및 인터페이스 변경 금지
* **하위 호환성을 깨는 변경 금지:** 팀원들과의 사전 합의 및 TRD(Technical Requirement Document) 수정 없이, 이미 기존 모듈이나 프론트엔드와 연결된 데이터베이스 컬럼명, 데이터 타입, 혹은 API 엔드포인트의 입출력 필드 명세를 독단적으로 변경하는 행위를 금지합니다.
* **마이그레이션 없는 DB 직접 수정 금지:** 로컬이나 스테이징 데이터베이스의 테이블 구조를 GUI 툴 등으로 직접 수정하는 행위를 금지합니다. 모든 스키마 변경은 반드시 추적 가능한 마이그레이션 스크립트 코드로 작성되어 형상 관리에 포함되어야 합니다.

### 3.4. 형상 관리 및 배포 프로세스 방해 금지
* **검증 없는 메인 브랜치 직접 푸시 금지:** 원격 저장소의 메인 브랜치(`main` 또는 `master`)에 직접 코드를 푸시(Direct Push)하거나 강제 푸시(`git push -f`)하는 행위를 금지합니다. 모든 코드는 작업 브랜치에서 보호된(Protected) 브랜치로 Pull Request(PR)를 통해서만 반영되어야 합니다.
* **코드 리뷰 및 자동화 검증 우회 금지:** 린트 에러(`eslint-disable` 남용), 빌드 실패, 혹은 테스트 실패가 발생했음에도 이를 수정하지 않고 검증 단계를 강제로 우회(bypass)하여 병합하는 행위를 금지합니다. 최소 1인 이상의 팀원 승인(Approve) 없이는 코드를 병합할 수 없습니다.

---

## 4. 검증 명령 (Verification Commands — 끝났는지 확인하는 방법)
개발을 완료한 후, 코드가 정상적으로 작동하고 규칙을 준수했는지 확인하기 위해 아래 명령어를 순서대로 실행해야 합니다.

**코드 스타일 및 정적 분석 검증:**
  ```bash
  npm run lint       # 코드 스타일 및 린트 규칙 검사
  npm run format     # 코드 포맷팅 상태 확인
  npm run build      # 프로젝트 빌드 정상 수행 여부 및 타입 체크 (TypeScript 사용 시)
  npm run test       # 단위 테스트 및 통합 테스트 전수 실행
  ```
